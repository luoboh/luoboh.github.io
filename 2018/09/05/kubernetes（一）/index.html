kubernetes（一）

发表于 2018-09-05 | 分类于 kubernetes

kubernetes名词解释

kubernetes组件

kubectl

这是 kubernetes 提供的客户端程序，也是目前常用的和集群交互的方式。创建、查看、管理、删除、更新 pod、service、replication controller 都行，还有更多其他命令，可以查看帮助文档。

etcd集群

etcd 是 kubernetes 存放集群状态和配置的地方，这是集群状态同步的关键，所有节点都是从 etcd 中获取集群中其他机器状态的，集群中所有容器的状态也是放在这里的。可用来搭建高可用集群，类似zookeeper，kubernetes 在 etcd 中存储的数据在 /registry 路径下。

Master组件

kubernetes 是典型的 master-slave 模式，master 是整个集群的大脑，负责控制集群的方方面面。

API Server

对外提供 kubernetes API，也就是 kubernetes 对外的统一入口。封装了 kubernetes 所有的逻辑，通过 RESTful 的方式供客户端使用，kubectl 就是最常用到的客户端程序。kubernetes内部所有模块也通过API　Server进行交互，各个模块通过APIServer将信息存入etcd，当需要获取这些数据或信息时，则通过API Server提供的REST接口来实现。　　

Scheduler

调度器：实现容器调度的组件，调度算法可以由用户自己实现。Scheduler 会收集并分析当前系统中所有 slave 节点的负载情况，在调度的时候作为决策的重要依据。
调度器监听 etcd 中 pods 目录的变化，当发现新的 pod 时，会利用调度算法把 pod 放到某个节点进行部署。可能的 scheduler 包括：
• random：随机调度算法
• round robin：轮询调度

Controller Manager Server

kubernetes所有资源对象的自动化控制中心，负责集群内的node、pod副本、服务端点（endpoint）、命名空间（namespace）、服务账号（service account）、资源定额（resource quota）等的管理，比如当某个node宕机时，controller manager会及时发现此故障并执行自动化修复流程，确保集群始终处于预期的工作状态。
包括Replication Controller，Node Controller，ResourceQuota Controller，Service Controller，Endpoint Controller等多个Controller，每一个Controller都负责一种具体的控制流程，并且由Controller Manager管理。

Slave组件

kubelet

kubelet 是 slave 上核心的工作进程，负责容器和 pod 的实际管理工作（创建、删除等）以及和 master 通信，内容包括：
• 负责容器的创建、停止、删除等具体工作，也包括镜像下载、 volume 管理
• 从 etcd 中获取分配给自己的信息，根据其中的信息来控制容器以达到对应的目标状态
• 接受来自 master 的请求，汇报本节点的信息

kube-proxy

正如名字提示的一样，kube-proxy 为 pod 提供代理服务，用来实现 kubernetes 的 service 机制。每个 service 都会被分配一个虚拟 ip，当应用访问到 service ip 和端口的时候，会先发送到 kube-proxy，然后才转发到机器上的容器服务。

kubernetes相关概念

Pods

pods由一个或多个container组成，共同完成一个应用，pod的容器必须在同一个物理机上，共享cpu和内存，pod时动态的，可以被删除和创建，若要持久性保存pod信息，可以通过共享volume实现。同一个pod中的容器共享网络namespace,有相同的IP和端口。目前kubernetes的业务可以 分为长期伺服型（long-running），批处理型（batch）、节点后台支撑型（node-daemon）和有状态应用型（stateful application），分别对应的控制器为deployment、job、daemonSet和StatefulSet。Pod里的容器可以是docker容器，rkt容器甚至是virtlet管理的虚拟机。
podIP+container port=endpoint
kubernetes中的event，是一个事件的记录，可以通过kubectl describe pod XXX来查看pod的event，用以定位问题，排除故障
pod上可以配置计算资源，cpu和memory，以千分之一的cpu为最小单位，记为m，比如cpu配额定义为100-300m，表示0.1~0.3个cpu，是绝对值，memory配额也是绝对值，单位是内存字节数。

Replication Controller

此处的RC与Controller Manager中的Replication Controller不同，此处的RC是一种kubernetes的一种资源对象，是Deployment的前身，而Controller Manager中的Replication Controller又叫做副本控制器，用于自动控制集群中的pod副本数量。
RC 保证在任意时刻都有指定数量的 pod 副本在运行。比如如果我们创建了一个 rc，指定某个 pod 运行 3 份：
刚开始的时候，rc 发现集群中没有这个 pod，它会创建 3 个 pod，
当某个 pod 不响应或者被删除了，它会检测到这个变化，然后新建一个正常的 pod（替换错误状态的 pod），保证系统有 3 个 pod 在运行
当错误的 pod 回复正常，或者用户又手动添加了几个 pod，rc 也会检测到，它会通过删除 pod 来保证只有 3 份副本在运行，但只适用于长期伺服型应用，比如web服务。
创建 replication controller 的时候，需要几个参数：
pod 的模板：用来创建应用的 pod 模板，在需要的时候根据这个模板创建 pod
replicas 副本数：要达到的目标状态，pod 需要运行几份
label：需要监控的 pod 的标签，通过比较 replicas 变量的值和标签实际发现的 pod 数量确定是否在目标状态

Service

service，也就是服务，是实际环境中对外提供服务的业务抽象。一个服务后端可能有多个容器提供支持，但是对外通过服务这一层只有一个接口，通过转发用户的请求到后端的应用上来实现。这样做的好处是：对外隐藏了后端的细节，使得扩展和维护变得更容易。
在 kubernetes 中，service 后面对应的是 pod，这种对应关系是通过 label 来匹配的。每个 service 会配置 label selector 对后端的 pod 进行选择，因此这种关系是动态的，pod 的添加和删除都会自动被关联进来。

当service发布出去后，kube-proxy相当于个软件负载均衡器，负责把对service的请求转发到后端某个pod实例上，并在内部实现服务的负载均衡和会话保持机制，但service不共用一个负载均衡器的IP，每个service分配了一个全局唯一的虚拟IP，成为clusterIP，这是一个伪造的IP，clusterIP无法被ping，因为没有实体网络对象相应，clusterIP+service port可以实现集群内部对service的访问，无法在外部使用，在service.yaml文件中，定义为NodePort，在所有node上打开同一个端口，便可通过集群任意节点IP+端口的形式在外部对集群进行访问；
此外，还要解决外部访问service的负载均衡问题，假设集群有10个node，外部请求只需访问负载均衡器的IP，由负载均衡器将流量转发到后端某个node的NodePort上，通常采用单独的组件实现负载均衡的功能，例如HAProxy或者Nginx

Labels

labels 就是标签的意思，主要用于过滤 pod、service 或者 replication controller，尤其是当集群的规模比较大的时候。
一个 label 就是一个键值对，比如 app：myweb，release：stable，用来表示用户自己定义的属性。给 pod 添加添加合理的标签会做到事半功倍的效果，增加可读性（通过 label 就能大致了解 pod 的功能）的同时也能方便查找对应的服务（service 和 replication controller 就是通过 label 来绑定后端 pod 的）。
Labels应用场景：

  1. kube-controller进程通过rc上定义的label selector来筛选要监控的pod副本数量，实现pod副本数量始终符合预期设定的全自动控制流程
  2. kube-proxy进程通过service 的label selector选择对应的pod，自动建立起每个service到对应pod 的请求转发路由表，实现service的智能负载均衡机制
  3. 对node定义特定的label，在pod的定义文件中使用NodeSelector这种标签调度策略，kube-scheduler进程可以实现pod的定向调度。

**podIP、nodeIP、clusterIP

在kubernetes中创建pod，会有一个podIP，nodeIP（表示pod所在的node），发布服务会有一个serviceIP，创建成功后会有个clusterIP，service服务默认是clusterIP类型，此时的clusterIP只能内部访问使用，如果service Type是NodePort，同时给service指定一个nodePort的值，nodePort开放的是节点的端口，这个值的默认范围是30000-32767（这个值在API server的配置文件中，用–service-node-port-range定义），集群外就可以使用kubernetes任意一个节点的IP加上nodePort端口访问该服务了。创建pod指定应用程序的端口，比如mysql默认开放3306端口，web应用默认开放8080端口，这些端口都在pod上。

kubernetes集群高可用

高可用性的kubernetes集群，应用或者服务的高可用就是pod的高可用，这个副本控制器可以实现，master和node节点高可用需要由etcd实现，etcd需要是高可用集群，部署多个master节点，再加个负载均衡组件负责流量的转发，还需要一个全局网络组件负责集群通信（比如Flannel）。
